# WeatherON 디자인 정체성 가이드

> 이 문서의 목적: OS(iOS/Android) 디자인 트렌드가 바뀔 때마다 앱 디자인이 끌려다니지 않도록, **WeatherON 고유의 디자인 축을 고정**하고 OS 트렌드는 그 축에 맞춰 "가져오는" 것임을 분명히 한다. 반대 방향(OS가 바뀌면 브랜드가 따라간다)은 금지한다.

---

## 1. 왜 이 문서가 필요한가

기획서 20장(UI 디자인 가이드)은 현재 iOS 26 Liquid Glass를 채택한 **구현 스펙**이다. 하지만 Liquid Glass는 영원하지 않다 — Apple과 Google은 몇 년 주기로 디자인 언어를 바꾼다(Skeumorphism → Flat → Material → Material You → Liquid Glass …). 기획서 20장만 보고 작업하면, 다음 OS 디자인 전환기에 "WeatherON다움"이 무엇이었는지 기준이 사라지고 매번 새 트렌드를 그대로 이식하게 된다.

이 문서는 **OS 트렌드보다 수명이 긴 것**만 담는다. Liquid Glass 같은 구현 디테일은 여기 두지 않고 기획서/CI·BI 문서에 둔다. 이 문서가 다루는 것은 "OS가 무엇으로 바뀌어도 WeatherON이 WeatherON처럼 보이게 만드는 것"이다.

---

## 2. 변하지 않는 축 (브랜드가 소유하는 디자인)

OS 트렌드와 무관하게 항상 고정되어야 하는 요소. 새 OS 디자인 언어를 받아들일 때도 아래는 절대 희석되지 않아야 한다.

**컨셉 — Quiet Horizon**: 정보를 나열하는 화면이 아니라 "하루를 켜는 스위치"라는 자기 인식. 모든 디자인 결정은 "이게 정보를 더 보여주는가, 결정을 더 빠르게 만드는가"로 검증한다 ([[product_design_philosophy]] 참고 — "결정을 제안" vs "날씨 수치 제공").

**브랜드 컬러 페어링 — Clear Navy + Warm Sun**: 히어로, 앱 아이콘, 워드마크, 1차 CTA, 온보딩 같은 "브랜드 모먼트"에만 사용. 기존 Navy(#0C1F3F)/Gold(#F0A020)는 신뢰감은 강하지만 금융·보안 앱처럼 무겁고 물먹은 인상으로 보일 수 있어 MVP 전까지 재검토 대상으로 둔다. 2026-06-25 이후 목업 기준은 Clear Navy(#10243F/#1F4E79)와 Warm Sun(#F4B63F)을 사용했고, 2026-07-09 이후 앱 페이지 배경은 더 가볍게 보이도록 Clear Navy를 #17365D/#2A5D8F 축으로 상향한다. Gold 계열은 브랜드 대표색보다 햇빛·주의·강조 의미색으로 축소한다. 라이트모드에서는 탁한 브라운 골드 대신 Dawn Orange(#C2410C)를 사용해 CTA와 활성 상태가 더 선명하게 보이도록 한다.

**기능색 — Sky/Clear-teal/Rain-red/Mist**: 날씨 상태·기능 카테고리를 나타내는 용도로만 사용하고, 브랜드 모먼트로 침범하지 않는다. 이 2트랙(브랜드색 vs 기능색) 분리 자체가 OS 트렌드와 무관하게 유지되는 규칙이다.

**공유 글리프 — 토글+태양노브**: 아이콘의 ON 스위치 형태와 워드마크 "O" 글리프가 같은 모티프(골드 원+광선)를 공유한다. 이 형태는 OS 아이콘 마스킹 규칙(원형/스쿠모픽/Liquid Glass 레이어링 등)에 맞춰 외피만 바뀌고, 내부 모티프는 유지된다.

**타이포 — Plus Jakarta Sans / Noto Sans KR / DM Mono / Manrope(워드마크 전용)**: OS 기본 시스템 폰트(SF Pro, Roboto 등)로 전면 교체하지 않는다. 시스템 폰트는 성능/접근성상 일부 시스템 UI(키보드, 알림 등)에서만 자연스럽게 노출되는 영역에 한해 허용.

**제품 철학 — "5초 안에 오늘 준비 끝"**: 인터랙션 깊이, 정보 위계, 기본 화면 구성의 기준점. OS가 새로운 인터랙션 패턴(제스처, 위젯 시스템 등)을 내놓아도, 이 패턴이 "5초 결정"을 더 빠르게 만드는지가 도입 여부의 1차 기준이다.

## 3. OS 트렌드를 받아들이는 원칙 (변할 수 있는 것)

OS 디자인 언어가 바뀔 때 받아들여도 되는 영역과, 받아들이는 방식의 원칙.

**받아들여도 되는 영역 — "구조적 chrome"**: 탭바, 모달/바텀시트, 시스템 컨트롤(토글, 슬라이더), 전환 애니메이션의 물리 법칙(스프링 이징 등), 레이어링/블러 같은 표면 처리 기법. 이런 요소는 OS와 이질감 없이 맞춰가는 것이 사용자 경험상 유리하다.

**받아들이는 방식 — 항상 브랜드 토큰을 통과시킨다**: OS 트렌드를 그대로 복제하지 않고, 반드시 WeatherON의 컬러/타이포 토큰으로 리틴트한 뒤 적용한다. 예: Liquid Glass의 블러+반투명 자체는 가져오지만, 틴트 컬러는 시스템 기본 백색이 아니라 Navy 틴트(`rgba(11,30,61,…)`)를 쓴다 — 기획서 20장에 이미 이렇게 명시되어 있다. 이것이 이 원칙의 실제 사례다.

**받아들이지 않는 경계**: 브랜드 모먼트(2장에 명시된 요소)를 직접 대체하거나 희석시키는 OS 트렌드는 받아들이지 않는다. 예를 들어 OS가 "전체 UI를 시스템 다이내믹 컬러로 통일"을 강제하는 방향으로 가더라도, 앱 아이콘·워드마크·CTA의 Navy+Gold는 고정한다.

**플랫폼 분기 허용**: iOS와 Android가 다른 디자인 언어를 쓰는 시점에는 구조적 chrome 레이어에서 플랫폼별로 분기해도 된다(iOS=Liquid Glass 스타일 블러, Android=Material 스타일 elevation 등). 단, 브랜드 모먼트는 두 플랫폼에서 동일하게 유지한다.

**Project Wind 분리 원칙**: `docs/Project Wind/`는 WeatherON 출시 후 대규모 UI 업데이트를 위한 신규 디자인 시스템 연구 트랙이다. 바람의 탑 기반 반응형 표면, 빛, 밀도, 모션 언어를 탐색하되, 2장의 고정 축(Quiet Horizon, Clear Navy + Warm Sun, 기능색, 공유 글리프, 5초 결정)을 희석하지 않는다. 현행 MVP/v1.0 구현 스펙은 `docs/design/WeatherON_UI_Design_Spec.md`를 우선한다.

## 4. 현재 적용 사례 (수시 교체되는 섹션)

> 아래는 "현재 시점의 구현"이며, 이 섹션만 OS 트렌드 전환 시 교체된다. 위 1~3장은 그대로 유지.

- **2026년 기준 적용 트렌드**: iOS 26 Liquid Glass — 구체적 블러/투명도/그라디언트 스펙은 기획서 `WeatherON_planning_v5.html` 20장에 정의.
- **Android 대응**: `backdrop-filter` 미지원 대응(RenderScript/BlurMaskFilter 폴백), 저사양 기기용 "투명도 줄이기" 토글 — 기획서 20장 구현 주의사항 참고.
- **아이콘 레이어링**: G-1 아이콘의 L1(불투명 Navy)/L2(크림 트랙+Navy 바)/L3(골드 노브+광선) 3-레이어 구조 — `brand/weatheron_ci_bi.html` 참고.
- **라이트/다크 모드 & 날씨 배경 애니메이션** (2026-06-20 추가): 인앱 UI 컬러 토큰의 라이트/다크 매핑, 홈(H1) 배경의 날씨 상태별(맑음/흐림/비/눈/밤/폭풍) 다이나믹 파티클·모션 스펙 — 기획서 `WeatherON_planning_v5.html` 20장에 정의. Gold 액센트와 기능색(Sky/Clear/Rain-Red)은 2장 원칙에 따라 테마 불변. 런처 아이콘의 다크 변형은 `brand/weatheron_ci_bi.html` 05장 참고.
- **온보딩 일러스트 방향** (2026-06-24 추가): O2/O6 일러스트는 Hybrid Chrome의 딥 네이비, 민트, 스카이 블루, 웜 골드와 맞는 글로시 카드 질감을 사용하되, 최종 출시본에서는 실제 앱 UI 스크린샷처럼 보이지 않도록 더 추상화된 시나리오 일러스트로 정리한다. O2는 4대 기능을 한 장면에 묶는 방향을 유지하고, O6는 목적지 경로와 목적지 알림 메시지가 좁은 카드 안에서도 잘리지 않도록 단순 구도 재제작을 우선 검토한다.
- **핵심 화면 시각·UX 고도화 기준** (2026-06-25 추가): 1차 MVP 목업 품질 기준은 A/O/H/C/G/P 핵심 플로우에서 먼저 고정한다. 모든 핵심 화면은 393×852 프레임 안에서 고정 CTA·탭바가 마지막 콘텐츠를 가리지 않아야 하며, 내부 스크롤바는 앱 화면에 노출하지 않는다. 카드/칩/CTA는 `WeatherON_design_system.jsx`의 반경·높이·상태 레이어 기준을 따르고, 검증용 상태 문구는 실제 앱 화면이 아닌 문서/플로우차트에만 유지한다. 코디·목적지·온보딩 이미지는 카드 슬롯 안에서 `object-fit: contain` 또는 고정 비율 슬롯으로 잘림 없이 표시한다.
- **Clear Weather UI 컬러 보정** (2026-06-25 추가, 2026-07-09 조정): WeatherON은 다크 전용 앱이 아니라 기기 설정을 따르는 라이트/다크 대응 앱으로 설계한다. 라이트 모드는 Light Background(#F5F9FC), Surface(#FFFFFF), Elevated(#EAF3FA), Primary Navy(#1F4E79), Text Primary(#142033)를 기준으로 하고, 다크 모드는 Dark Background(#17365D), Surface(#1F4E79), Elevated(#214A78), Text Primary(#F8FBFF), Text Secondary(#B9CBE0)를 기준으로 한다. 다크 기능색은 Fresh Sky(#4AA3DF), Clear Mint(#2FC6A3), Warm Sun(#F4B63F), Rain Red(#E97F77)를 사용한다. 라이트 기능색은 Dawn Orange(#C2410C), Clear Sky(#237BBD), Action Teal(#007F73), Coral Warm(#C84A2F)을 사용해 흰 카드 위에서도 포인트가 흐려지지 않게 한다. 저투명 보조 텍스트는 4.5:1 대비 위험이 있으므로 소형 텍스트는 0.78 이상 투명도를 기본값으로 둔다.
- **라이트/다크 목업 검증 방식** (2026-06-25 추가): 화면별 파일을 `Light/Dark`로 복제하지 않고, 프리뷰 상단의 테마 토글로 같은 화면을 전환해 검증한다. 현재 프리뷰 네비에 노출된 43개 화면은 공통 `WeatherON_theme_tokens.js`의 라이트/다크 배경·프레임 토큰을 우선 적용한다. 이후 화면별 카드, 칩, 이미지 슬롯의 세부 대비는 같은 토글 상태에서 추가 폴리싱한다. 이 방식은 화면 수 폭증을 막고 동일 플로우에서 대비·CTA·이미지 슬롯 문제를 바로 비교하기 위한 기준이다.
- **BI/CI 재검토 원칙** (2026-06-25 추가): 앱 로고와 워드마크는 1차 작업에서 즉시 교체하지 않는다. 다만 기존 Navy/Gold가 "맑음, 준비, 가벼움, 신뢰, 5초 결정"을 충분히 전달하는지 MVP 전 브랜드 검증 항목으로 둔다. 앱 아이콘·워드마크는 유지하되, 화면 UI는 더 밝고 선명한 Clear Weather UI로 먼저 검증한다.
- **브랜드 카피 원칙** (2026-06-25 추가): 사용자 노출 문구는 "결정을 대신한다", "정확도를 높인다", "국내외 전무"처럼 법무·신뢰 리스크가 큰 단정 표현을 피한다. 기본 표현은 "빠르게 정할 수 있게 돕는다", "기준을 세밀하게 맞춘다", "현재 조사 범위에서 차별성이 크다"처럼 실용적이고 검증 가능한 톤으로 유지한다.

### 4-1. 채택 항목 (Liquid Glass / Material — 3장 원칙 적용 사례, 2026-06-20 결정)

> 3장 "구조적 chrome" 범위 내에서, 각 OS 트렌드의 **메커니즘만** 가져오고 **항상 Navy/Gold/기능색 토큰으로 리틴트**한 것. `WeatherON_H1_hybrid_chrome_DEMO.jsx`에 구현됨.

**Liquid Glass에서 채택**
- 바텀시트/모달의 블러+반투명 레이어링 — 시스템 백색 대신 Navy 틴트(`rgba(12,31,63,0.55)`) + `backdrop-filter: blur(20px)`.
- 전환 애니메이션의 스프링 물리(오버슈트 이징, `cubic-bezier(0.34,1.56,0.64,1)`) — 시트 등장/탭 전환에 적용.
- 연속 곡률(squircle) 코너 — 카드/시트 모서리에 한정 적용. 정보 전달 요소가 아니므로 브랜드 희석 없음.
- 스페큘러 하이라이트는 **일반화하지 않음** — 공유 글리프(토글+태양노브) 자체에만 한정.

**Material Design에서 채택**
- 바텀시트 드래그 핸들 제스처 어포던스 — 사용자에게 익숙한 관습이라 구조적 chrome으로 인정.
- 상태 레이어(state layer) 프레스 피드백 — Material 기본 리플의 "퍼지는 원" 애니메이션은 배제하고, Gold/기능색 톤다운 틴트 오버레이(opacity 0→0.12, 120ms)로 대체.
- 엘리베이션 "체계"만 채택 — Material 톤 팔레트 대신 Navy 톤 스텝(PANEL → PANEL_L1 → PANEL_L2)으로 깊이감 표현.

**받지 않은 것 (2장 경계 침범으로 판단, 명시적 배제)**
- Material You 다이내믹 컬러(월페이퍼 추출 컬러).
- 각 트렌드의 디폴트 틴트(시스템 백색/그레이) — 항상 Navy 틴트로 교체.
- FAB의 정형화된 원형 실루엣/엘리베이션 티어 시스템.
- 리플의 문자 그대로의 원형 확산 애니메이션.

## 5. 다음 OS 트렌드 전환기 체크리스트

새로운 OS 디자인 언어(또는 대규모 개편)가 나왔을 때, 받아들이기 전에 점검할 질문:

1. 이 트렌드가 브랜드 모먼트(2장)의 컬러·글리프·타이포를 직접 대체하려 하는가? → 그렇다면 거부하거나 토큰 리틴트로 흡수.
2. 구조적 chrome에만 영향을 주는가? → 받아들이되 Navy/Gold/기능색 토큰을 통과시켜 적용.
3. "5초 안에 오늘 준비 끝"이라는 결정-우선 경험을 더 빠르게/느리게 만드는가? → 빠르게 만들면 적극 도입, 느리게 만들면 거부.
4. iOS/Android 동시 지원이 가능한가, 플랫폼 분기가 필요한가?
5. 접근성(대비, Reduce Transparency 등) 요구사항을 해치지 않는가?
