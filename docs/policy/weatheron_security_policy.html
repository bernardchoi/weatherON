00 · Principles
보안 설계 4대 원칙
WeatherON의 보안 정책은 개별 기능 단위가 아니라 "무엇을 수집하지 않을 것인가"를 먼저 정의하고, 그 위에 인증·전송·저장 계층의 보호 장치를 쌓는 순서로 설계한다. 아래 4대 원칙은 이후 모든 섹션의 구현 결정 기준이 된다.
📦

Data Minimization

실시간 위치, 정확 주소, 코디 사진 등은 서버에 저장하지 않는다. 위치는 반경 단위로 가공해 처리하고 원본 좌표는 즉시 폐기한다.

🔐

Privacy by Design

기기 언어 자동 적용 후 스플래시와 Guest 홈을 기본값으로 두고, 계정은 저장·동기화·히스토리·알림 확장 등 부가가치에만 결합한다.

🛡️

Defense in Depth

토큰 저장(Keychain/Keystore), 전송 암호화(TLS+Pinning), 저장 암호화(로컬DB), 서버 접근제어(Firestore Rules)까지 계층마다 독립적 방어선을 둔다.

📜

Compliance by Default

개인정보보호법·위치정보법·GDPR·App Store/Play 정책 요구사항을 사후 대응이 아닌 설계 시점부터 매핑해 반영한다.

01 · Authentication
인증 보안 — OAuth 2.0 + PKCE & 토큰 관리
모든 로그인은 OAuth 2.0 Authorization Code Flow with PKCE를 표준으로 사용한다. Provider(카카오·네이버·LINE·Google·Apple)의 토큰은 클라이언트에 전달하지 않고 백엔드에서만 교환·보관하며, 앱은 Firebase Auth session/custom token 기준으로 로그인 상태를 유지한다. 세부 구현 기준은 docs/architecture/WeatherON_MVP_기술_ADR.md를 따른다.
PKCE 인증 흐름
1
code_verifier / code_challenge 생성
로그인 버튼 탭 시 앱이 랜덤 문자열 code_verifier를 생성하고, SHA-256으로 code_challenge를 계산해 메모리에 보관한다.
2
Authorization 요청
code_challenge + code_challenge_method=S256을 포함해 Provider 인증 화면(시스템 브라우저 / ASWebAuthenticationSession / Custom Tabs)으로 이동한다.
3
사용자 로그인 & 동의
카카오/네이버/LINE/Google/Apple 화면에서 로그인 후 요청 스코프(이메일, 프로필 등)에 대한 동의 여부를 사용자가 직접 선택한다. 이메일 코드는 별도 인증 화면에서 1회용 코드를 확인한다.
4
Authorization Code 반환
Provider가 등록된 Redirect URI(Universal Link / App Link / Custom Scheme)로 1회용 authorization code를 전달한다.
5
백엔드 토큰 교환
앱은 code + code_verifier를 백엔드 /auth/exchange로 전송. 백엔드가 Provider 토큰 엔드포인트에서 access/id token을 받아 사용자 식별(sub)을 확인한다. Provider 토큰은 클라이언트로 반환되지 않는다.
6
Firebase 세션 생성
Cloud Functions가 검증된 user ID로 Firebase Custom Token을 발급하고, 앱은 Firebase Auth로 로그인 세션을 생성한다. Provider 토큰은 클라이언트로 반환하지 않는다.

자동 추천 기반 노출 원칙
앱은 첫 실행 시 로그인 선택 화면을 강제하지 않고, 기기 언어 자동 적용과 스플래시 후 Guest 홈으로 바로 진입한다. 사용자가 코디 저장, 옷장 저장, 장소 저장, 알림 추가, 여행 저장, 프리미엄 진입처럼 계정이 필요한 기능을 탭했을 때만 계정 연결 화면을 호출한다. UI 언어는 OS/기기 언어를 우선 적용하고 설정에서 변경할 수 있게 하며, 로그인 방법 추천은 현재 여행 위치보다 계정 환경 신호를 우선해 별도로 판단한다. 우선순위는 스토어 국가, SIM/전화 국가, OS 지역, 기기 언어, 타임존, 현재 위치 순서이며, 한국은 카카오·네이버를 상위 노출하고 일본은 LINE을 상위 노출한다. 일반 해외는 Google·Apple·이메일 코드를 기본으로 제공한다. 예를 들어 일본어 기기·일본 스토어 사용자가 한국 여행 중 가입하면 현재 위치가 한국이어도 LINE/Apple/Google을 우선 추천한다. 국가/권역 선택은 회원가입의 선행 단계로 두지 않으며, 계정 연결 화면에도 국가/지역 선택 버튼을 노출하지 않는다. 로그인 수단 추천은 내부 신호로 자동 처리하고, "다른 방법 보기"는 추가 로그인 수단을 펼치는 용도로만 사용한다. 중국권 등 Google/Apple/LINE 접근성이 제한되거나 실명·데이터 규제가 큰 특수 환경 국가는 1차 제공 범위에서 제외하고 별도 보안·법무 검토 후 진출 여부를 결정한다.
추가 인증 보호 장치

🔑 생체 인증 (App Lock)

  • • Face ID / Touch ID — 앱 진입 시 선택적 잠금 (설정 ON 시)
  • • 생체 인증은 OS 레벨에서 처리, 생체 데이터 자체는 앱/서버에 전달되지 않음
  • • 생체 인증 실패 시 디바이스 PIN/패턴으로 Fallback

⏱️ Rate Limiting & Lockout

  • • 로그인/인증 시도 5회 실패 시 30분 잠금 (IP + 계정 단위 병행 적용)
  • • Firebase 세션 갱신 및 이메일 코드 검증 요청도 동일 Rate Limit 적용
  • • 비정상 반복 요청은 Cloud Armor / Firebase App Check로 추가 차단

📱 세션 / 디바이스 관리

  • • 로그인된 기기 목록 조회 (OS, 모델, 최초 로그인 시각, 마지막 활동)
  • • 특정 기기 원격 로그아웃 → 해당 Firebase 세션 즉시 revoke
  • • 신규 기기 로그인 시 푸시/이메일로 알림 (디바이스 바인딩, §03 참조)

👤 Guest Mode 경계

  • • 비회원도 현재 위치 날씨·코디·우산 추천, 외출 알림 1개, AI 추천 1~2곳 이용 가능
  • • 코디 저장, 옷장 저장, 장소 저장, 알림 추가, 여행 저장, 프리미엄 진입 시에만 OAuth 플로우 호출 — 첫 화면 강제 가입 없음
  • • Guest 식별자(디바이스 단위 UUID)는 서버 분석용으로만 사용, 광고 ID와 분리
02 · Transport
통신 보안 — TLS, Certificate Pinning, API 보호
모든 API 통신은 HTTPS 강제(HTTPS Only)이며, 추가로 SSL Certificate Pinning을 적용해 신뢰할 수 없는 인증서 체인을 통한 중간자(MITM) 공격을 방어한다.
전송 계층 강제 설정

🍎 iOS — App Transport Security

  • • NSAllowsArbitraryLoads = false — 모든 도메인 HTTPS만 허용
  • • 최소 TLS 1.2, 권장 TLS 1.3
  • • 예외 도메인(NSExceptionDomains) 등록 금지 — 서드파티 SDK도 HTTPS 엔드포인트만 허용

🤖 Android — Network Security Config

  • • cleartextTrafficPermitted="false" 전역 적용
  • • base-config에 trust-anchors를 시스템 CA로 제한 (사용자 추가 CA 신뢰 안 함)
  • • 디버그 빌드에서만 별도 debug-overrides로 로컬 프록시 허용
Certificate Pinning 구현
현재 인증서와 차기 교체 예정 인증서(Backup Pin)를 함께 등록해, 인증서 갱신 시점에 앱 강제 업데이트 없이도 Pinning이 끊기지 않도록 한다. 핀은 SPKI(Public Key)의 SHA-256 해시 기준으로 등록한다.
// Android — OkHttp CertificatePinner val certificatePinner = CertificatePinner.Builder() .add("api.weatheron.app", "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=") // 현재 인증서 .add("api.weatheron.app", "sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=") // 백업 핀(차기 인증서) .build() OkHttpClient.Builder() .certificatePinner(certificatePinner) .build()
// iOS — TrustKit 설정 (Info.plist) TSKConfiguration = { TSKSwizzleNetworkDelegates: true, TSKPinnedDomains: { "api.weatheron.app": { TSKPublicKeyHashes: [ "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=", // 현재 인증서 "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=" // 백업 핀 ], TSKEnforcePinning: true } } }
Pinning 실패 시: 요청을 즉시 중단하고 사용자에게 "보안 연결을 확인할 수 없습니다" 안내 + 백그라운드에서 보안 로그 전송(원격 Pin 업데이트 트리거)
API 요청 보호

🧾 요청 인증

  • • 모든 API 요청 헤더에 Authorization: Bearer <Firebase ID Token> 필수
  • • 클라이언트 호출은 Firebase App Check 토큰을 함께 첨부해 비정상 클라이언트(에뮬레이터/스크립트) 차단

♻️ Replay 방지

  • • 민감 요청(계정 삭제, 결제 등)에는 요청 타임스탬프 + nonce 포함, 서버에서 5분 이내 + 1회성 검증
  • • 동일 nonce 재사용 시 요청 거부 및 보안 로그 기록
03 · Device & Data
디바이스 & 데이터 보안
토큰·로컬 캐시 등 디바이스에 남는 모든 데이터는 OS 보안 저장소를 거치고, 위치·사진 등 민감 데이터는 가능한 한 디바이스에서만 처리하고 서버로 보내지 않는다.
로컬 저장소 암호화

🍎 iOS Keychain

  • • 저장 옵션: kSecAttrAccessibleWhenUnlockedThisDeviceOnly
  • • iCloud Keychain 동기화 비활성화 — 토큰은 발급된 기기에만 존재
  • • UserDefaults / 파일 시스템에 토큰·PII 저장 금지

🤖 Android Keystore

  • • AES-256-GCM 키를 StrongBox / TEE에서 생성·보관
  • • Firebase Auth 세션과 로컬 민감값은 EncryptedSharedPreferences(Jetpack Security)로 암호화 저장
  • • AsyncStorage 류 평문 저장소 절대 금지

🗄️ 로컬 캐시 DB

  • • 날씨/코디 히스토리 캐시는 SQLCipher(AES-256) 또는 Realm 암호화 적용
  • • 암호화 키는 Keychain/Keystore에 보관된 키로 래핑(key wrapping)
  • • 캐시 만료 정책: 날씨 캐시 최대 1시간, 코디/사진 캐시 최대 7일 후 자동 삭제

📸 코디 사진

  • • 촬영/선택한 코디 사진은 온디바이스 처리만, 서버 업로드·저장 없음
  • • AI 추천에 사용되는 경우에도 이미지 자체가 아닌 온디바이스 추출 태그만 전송
위치 데이터 처리 파이프라인
1
권한 — "사용 중에만 허용" 권장
위치 권한은 항상 허용(Always)이 아닌 "앱 사용 중" 범위로 요청. 백그라운드 위치 수집 없음.
2
원본 좌표 — 메모리 한정
GPS 원본 위도/경도는 디바이스 메모리에서만 사용, 디스크에 기록하지 않고 화면 갱신 후 즉시 폐기.
3
반경/Geohash 변환
서버 요청 전 Geohash precision 5(약 4.9km 격자) 또는 행정구역 코드로 변환 — 개인 특정이 불가능한 단위로 가공.
4
서버 전송 & 미저장
가공된 위치 단위로만 날씨/추천 API 호출. 서버는 요청 처리 후 위치 값을 로그·DB에 영속 저장하지 않음.
앱 바이너리 보호

🧩 난독화 / 심볼 제거

  • • Android: R8/ProGuard — minifyEnabled true, shrinkResources true, 난독화 매핑 파일은 별도 보관(크래시 분석용)
  • • iOS: Release 빌드에서 디버그 심볼 strip, Bitcode 비활성화

🔓 루팅 / 탈옥 감지

  • • Android: RootBeer 등으로 su 바이너리·Magisk·비정상 빌드 태그 탐지
  • • iOS: IOSSecuritySuite로 Cydia/탈옥 파일 경로·심볼릭 링크 점검
  • • 감지 시 동작: 1차 경고 배너 노출 + 결제/계정 보안 기능(비밀번호 변경, 결제) 제한. 핵심 기능 전체 차단은 하지 않음(접근성 고려)

🐞 디버그 빌드 차단

  • • Release 빌드: debuggable=false, 개발자/테스트 메뉴 비활성화
  • • 로깅 레벨: Release 빌드에서 Verbose/Debug 로그 strip (Timber release tree no-op 등)
  • • 환경변수/엔드포인트는 빌드 타임 주입, 하드코딩된 운영 키 금지

🔔 디바이스 바인딩 & 이상탐지

  • • 최초 로그인 시 디바이스 핑거프린트(OS, 모델, 최초 로그인 시각) 등록
  • • 등록되지 않은 신규 기기에서 로그인 시 푸시/이메일로 즉시 알림
  • • 알림에는 "내가 아니라면" 즉시 전체 세션 종료 + 비밀번호/연결 계정 점검 안내 링크 포함
04 · Backend & Infra
백엔드 / 인프라 보안
Firebase 기반 인프라(Firestore, Cloud Functions)의 접근 제어, 제3자 API 키 관리, 로깅/모니터링 체계를 정의한다. 핵심 원칙은 "클라이언트를 신뢰하지 않는다" — 모든 요청은 서버 측에서 재검증한다. MVP 백엔드 구조는 Firebase Auth + Cloud Functions + Firestore + FCM + Secret Manager + App Check로 확정한다.
Firestore Security Rules
// firestore.rules — 사용자별 데이터 격리 rules_version = '2'; service cloud.firestore { match /databases/{database}/documents { match /users/{userId} { allow read, update, delete: if request.auth != null && request.auth.uid == userId; allow create: if request.auth != null && request.auth.uid == userId && request.resource.data.keys().hasOnly(['profile','settings','createdAt']); } // 코디 히스토리·여행 플래너 등 하위 컬렉션도 본인만 접근 match /users/{userId}/{collection}/{docId} { allow read, write: if request.auth != null && request.auth.uid == userId; } // 기본 거부 — 명시되지 않은 경로는 모두 차단 match /{document=**} { allow read, write: if false; } } }

표준 대응 흐름
1
탐지 (Detect)
Cloud Monitoring 알림, 사용자 신고, 외부 제보 등을 통해 인지. 담당자가 즉시 심각도를 1차 분류.
2
격리 (Contain)
유출 경로 차단 — 노출된 키 즉시 폐기/로테이션, 영향받은 토큰/세션 강제 무효화, 필요 시 관련 API 임시 중단.
3
영향 분석 (Assess)
영향받은 사용자 수·데이터 항목 파악(Cloud Logging 추적), Critical/High의 경우 법적 통지 의무 여부 판단.
4
통지 (Notify)
법정 기한 내 감독기관 신고 + 영향받은 사용자에게 이메일/푸시로 통지(노출 항목, 권장 조치 포함). 개인정보 보호책임자가 총괄.
5
복구 (Remediate)
근본 원인 수정(코드/설정), 추가 모니터링 룰 적용, 영향받은 사용자 비밀번호·연결 계정 재설정 안내.
6
사후 보고 (Postmortem)
사고 타임라인·원인·재발 방지 대책을 문서화하고 체크리스트(§07)에 반영. 내부 공유 후 보존.
법정 통지 기한 & 담당

⏰ 통지 기한

  • • 개인정보보호법 — 일정 규모 이상 유출 시 개인정보보호위원회/한국인터넷진흥원(KISA)에 24시간 이내 신고
  • • GDPR — 인지 후 72시간 내 관할 감독기구(DPA) 통지, 고위험 시 정보주체에게도 통지
  • • 사용자 통지는 이메일 + 앱 푸시 병행, 노출 항목과 권장 조치(비밀번호 변경 등)를 구체적으로 명시

👤 대응 조직

  • • 개인정보 보호책임자(CPO) — 통지·신고 총괄, 처리방침에 연락처 게시
  • • 침해 대응팀: 개발(원인 조치) · 법무(통지 의무 판단) · CS(사용자 응대) 3개 역할로 구성
  • • 비상 연락 체계: 1차 담당자 무응답 시 30분 내 2차 에스컬레이션
07 · Pre-Launch Checklist
출시 전 보안 체크리스트
MVP(Phase 1) 출시 전 반드시 확인해야 하는 항목을 영역별로 정리했다. 모든 항목은 위 섹션의 구현 내용과 1:1로 대응한다.

인증 / 계정

  • OAuth 2.0 + PKCE 플로우 5개 Provider(카카오/네이버/LINE/Google/Apple) 및 이메일 코드 인증 테스트 완료
  • Apple 로그인 포함 여부 — iOS 심사 거절 사유 1순위
  • Firebase Auth 세션과 로컬 민감값이 OS 보안 저장소에만 저장되는지 코드 리뷰로 확인 (AsyncStorage 미사용)
  • Firebase Admin SDK 세션 revoke 및 원격 로그아웃 동작 테스트
  • 로그인 실패 Rate Limiting(5회/30분) 동작 확인
  • Guest Mode에서 핵심 기능이 회원가입 없이 동작하는지 확인

통신 / 네트워크

  • iOS ATS 예외 도메인 없음, Android cleartextTrafficPermitted=false 확인
  • Certificate Pinning 적용 + 백업 핀 최소 1개 등록
  • TLS 1.2 이상 강제, TLS 1.3 지원 확인
  • 모든 API 요청에 App Check 토큰 첨부 확인

디바이스 / 데이터

  • 로컬 캐시 DB 암호화(SQLCipher/Realm) 적용 및 키 래핑 확인
  • 위치 정보가 Geohash/반경 단위로만 서버 전송되는지 네트워크 캡처로 검증
  • 코디 사진이 서버로 업로드되지 않는지 확인
  • 캐시 만료 정책(날씨 1시간/코디 7일) 동작 확인

앱 바이너리

  • Android R8/ProGuard 활성화, 매핑 파일 별도 보관
  • Release 빌드 debuggable=false, 디버그 메뉴 제거
  • 루팅/탈옥 감지 동작 및 경고 배너 노출 테스트
  • Release 로그에서 PII/토큰 값 미출력 확인

ON Square Lite 운영 안전

  • Weather Note 신고·사용자 차단·운영자 숨김 처리 동작 확인
  • 작성 빈도 제한·중복 게시·금칙어·스팸 탐지 규칙 적용
  • 정확한 위치 비노출, 도시/Geohash 단위 표시 및 24시간 만료 검증
  • 비회원 읽기 전용, 작성·리액션은 인증 사용자만 허용

백엔드 / 인프라

  • Firestore Security Rules — 에뮬레이터로 본인 외 접근 차단 테스트
  • 모든 Cloud Functions에서 context.auth 검증 적용
  • 제3자 API 키 — 패키지명/Bundle ID/SHA-1 제한 설정 확인
  • Gemini 등 민감 키가 클라이언트에 노출되지 않는지(디컴파일 점검) 확인
  • 로그에 이메일/위치 원본/토큰 값 미기록(PII 스크러빙) 확인

법무 / 정책
