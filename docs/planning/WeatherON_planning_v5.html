01 · Overview
이 앱이 필요한 이유

기존 날씨 앱은 수치를 보여줄 뿐, 결정을 내려주지 않는다. 강수확률 60%를 보고 우산을 챙길지 말지 사용자가 직접 판단해야 한다. WeatherON은 다르다 — 강수량·강수 지속시간·바람·온도·체감온도를 종합해 "오늘은 3단 우산, 트렌치코트, 방수 스니커즈"까지 직접 제안한다. 목적지 등록 없이도 설정한 시간에 날씨·코디·신발 알림을 받을 수 있고, 목적지를 등록하면 출발지↔목적지 날씨 비교, 소요시간 역산, 출발 10분 전 신발 알림까지 확장된다. 일상 알림은 기본, 목적지 기능은 선택 — 두 경우 모두 완결된 경험을 제공한다.

MVP Function PRD
MVP 1차 개발 기준은 docs/planning/WeatherON_MVP_기능_PRD.md를 따른다. 해당 PRD는 audit가 제안한 A/O/H/C/G1-G2/P/M/R 범위만 대상으로 하며, 날씨 API 연동, 코디 룰엔진, 우산·신발 추천, 옷장, 목적지 케어, 스마트 알림, MY/정책 상태를 실제 개발 가능한 기능 단위로 정리한다. 도보여행은 기존 WeatherON_도보여행_PRD.md를 별도 기준으로 유지한다.
Onboarding O2 Principle
O2 기능 소개는 회원가입 유도 화면이 아니라 개인화 온보딩을 납득시키는 설득 화면으로 둔다. 첫 화면에서 "날씨 보고, 입고, 나가세요" 메시지와 4대 기능(코디·우산·신발·출발 알림)을 한눈에 보여주고, 사용자가 기능 카드를 탭하면 각 기능의 무엇을 해주는지 / 언제 유용한지 / 다음 단계에서 필요한 설정을 짧게 보여준다. CTA는 추천 기준 설정하기로 표현해 O3 권한 요청과 O4 추천 기준 설정으로 이어지는 이유를 명확히 한다. O2 일러스트는 MVP 목업용으로는 현재 UI 컨셉과 잘 맞지만, 최종 출시 전에는 실제 앱 UI 스크린샷처럼 보이는 인상을 줄이고 더 추상화된 WeatherON 전용 시나리오 이미지로 polish한다.

Destination Care Principle
목적지 등록은 필수 가입 절차가 아니라 목적지 기준 알림을 세밀하게 만드는 선택 설정이다. 사용자가 회사·학교·공항·숙소 같은 자주 가는 곳을 1회 등록하면 출발지↔목적지 날씨 비교, 출발 시각, 신발·우산 알림 기준에 포함된다. 등록하지 않아도 현재 위치 기준 Mode A 알림은 그대로 작동한다. O6 일러스트는 목적지·경로·날씨 비교의 방향은 맞으나, 최종 출시 전에는 좁은 카드 안에서도 경로와 목적지 알림 메시지가 명확히 보이는 단순 구도로 재제작을 우선 검토한다.

기획서의 기존 도면형 가이드는 고정 조건이 아니라 초기 방향으로만 취급한다. Acloset·Whering·Stylebook·Weather Fit류 서비스의 공통 운영 방식처럼 "실제 옷장 기반 추천, 매일 아침 한눈에 판단, 시간대별 착장 변화, 셔플/대체 조합, 착용 기록"을 WeatherON 문맥에 맞게 적용한다. 홈(H1)은 4-piece 아이템 썸네일과 15°→26° 레이어 근거를 함께 노출한다. 코디 탭(C1)은 단순 색상 카드가 아니라 오버핏 가디건, 린넨 티셔츠, 슬랙스, 스니커즈처럼 개별 옷의 실루엣·구조가 보이는 아이템 카드 그리드와 한 줄 판단을 제공한다. MVP 목업은 생성형 PNG 의류 컷으로 표현하고, 실제 출시 단계에서는 사용자 옷장 사진 또는 배경 제거된 제품컷으로 교체한다. 우천 여부, 옷장 매칭률, 시간대별 착장 판단은 칩 수준으로 압축해 가독성을 우선한다. 임의 인물 사진이나 아바타 대신 사용자 옷장 기반 제품컷형 이미지를 사용해 성별·체형·국가권 편향을 줄인다.
C1/C4 흐름 원칙: C1은 오늘 코디를 빠르게 판단하는 요약 화면이고, 코디 카드 또는 "상세 보기"를 탭하면 C4 코디 상세로 이동한다. C1 상단에는 추천 상태 카드를 두어 옷장 매칭 수, 스타일 기준, 저장 계정 필요 여부를 짧게 보여준다. 기준 수정은 O4 스타일 기준으로 연결하고, 저장이 필요한 액션은 A2 계정 연결로 이동시켜 추천 확인과 저장 액션을 분리한다. C1에서 Guest가 코디 저장을 누르면 pendingAction=코디 저장, returnTo=C1, resumeSave=true를 유지하고 A3 완료 후 C1 저장 완료 상태로 복귀한다. C1의 셔플은 단순 장식 버튼이 아니라 outfitVariant를 토글해 요약 카드, 판단 카드, C4 상세 진입 상태가 같은 조합을 바라보게 한다. C4에서 재추천한 코디는 outfitVariant 상태로 C1 요약 카드와 H1 대표 코디까지 전달해, 상세에서 바꾼 코디가 목록이나 홈으로 돌아오며 기본 세트로 되돌아가지 않게 한다. C4는 코디 저장/공유와 대화형 재추천을 담당하며, 재추천 결과는 상단 코디 뷰에 즉시 반영한다. 상단 코디 뷰는 도면형 스펙보다 실제 의류 PNG 조합을 우선 노출하고, 08시/14시/18시 같은 시간대별 착용 판단과 각 아이템의 역할·아이템명·날씨 근거를 붙여 최종 MVP 목업 수준의 착장 이해도를 확보한다. C4에는 상세 상태 카드를 두어 재추천 반영 여부, 저장 전 상태, 공유 준비 상태를 표시하고 C1 복귀 CTA를 제공한다. 공유는 즉시 외부 공유로 넘기기보다 먼저 공유 준비 완료 상태를 보여주고, 실제 OS 공유 시트는 구현 단계에서 연결한다. 뒤로가기는 C1 요약 상태로 복귀한다. Guest 상태에서 C4 저장을 탭하면 A2 계정 연결 gate를 호출하고, A3 완료 후 C4 저장 완료 상태를 표시하며 C1 복귀 시 저장 완료 상태를 이어준다.
C1/C2/C3 옷장 흐름 원칙: C1의 상단 옷장 아이콘과 "내 옷장 관리" 행은 C2 옷장 관리로 연결한다. C2는 목록·필터·빈 상태를 함께 담당하고, 우상단 추가 버튼과 빈 상태 CTA는 C3 옷 등록으로 이동한다. C2에는 옷장 상태 카드를 두어 현재 필터의 아이템 수, Guest/계정 저장 상태, 추천 복귀, 옷 추가 CTA를 제공한다. Guest는 옷장 조회와 샘플 확인까지 가능하지만, 옷 추가·수정은 pendingAction=옷장 저장, returnTo=C3로 A2/A3 계정 연결 gate를 거친 뒤 C3 등록 화면에서 계속 진행한다. 아이템 탭 시 수정/삭제 시트를 열고, 수정은 C3 등록 패턴으로 이동하며 삭제 후에는 추천 기준을 즉시 재계산한다. C3는 사진과 카테고리를 필수로 받고 저장 완료 피드백 후 C2로 복귀한다. C3에는 등록 상태 카드를 두어 사진 선택, 카테고리, Guest/계정 저장 상태, 저장 가능/대기 상태를 표시하고, 사진 미선택 상태에서는 저장 버튼을 비활성화한다. 사진 선택 후에도 Guest 상태라면 저장 탭 시 사진/카테고리/이름/계절 상태를 보존하고 pendingAction=옷장 저장, returnTo=C3, resumeSave=true로 A2 계정 연결을 호출한다. A3 완료 후 C3 저장 완료 상태로 복귀하며 계정 연결 상태에서만 실제 옷장 저장과 추천 기준 반영을 완료한다. 옷장이 비어 있어도 기본 날씨 룰 추천은 유지하되, 첫 옷 등록 시 실제 보유 아이템 기반으로 추천 기준이 세밀해진다는 메시지를 제공한다.
H1 홈 연결 원칙: H1의 대표 코디 카드는 C1 코디 메인으로 연결해 Guest도 추천 상세를 바로 확인할 수 있게 한다. C1 셔플 또는 C4 재추천 후 홈으로 복귀하면 H1 대표 코디도 같은 outfitVariant를 받아 의류 컷, 제목, 날씨 근거 카피를 함께 갱신한다. H1의 우산 추천 카드는 현재 위치 또는 목적지에 비 신호가 있고 아직 확인하지 않은 경우에만 노출하는 실시간 의사결정 카드로 둔다. 사용자가 H4 우산 추천 상세를 열람하면 umbrellaSignalReviewed 상태를 남겨 H1에서는 같은 우산 카드를 숨기고, H3 알림 사이드바에는 우산 추천 이력을 남겨 재진입할 수 있게 한다. 따라서 H1은 "지금 바로 결정해야 하는 액션", H3는 "알림 센터/이력/딥링크 보관함" 역할로 분리한다. 홈에는 Guest/위치/알림 상태를 한 줄 요약으로 보여주어 현재 사용 가능 범위를 빠르게 인지하게 하고, 알림 설정 전 상태는 O3 권한 단계로 연결한다. 하단 탭은 홈 안에서 선택 상태만 바꾸지 않고 코디(C1), 출발(G1), MY 설정 허브(M1), 소셜(S1 ON Square)로 실제 화면 이동을 제공한다. 최초 소셜 진입 안내는 ON Square 내부 S0 상태로 처리한다. 목업 프리뷰에서는 로그인·권한·위치·프리미엄 같은 앱 전역 값은 Session State로 유지하고, 스타일 기준·핏·연령대·위치 관리 완료·스마트 알림 세부 토글처럼 설정에 남아야 하는 값은 Settings State로 유지한다. outfitVariant·selectedDestination·returnTo·pendingAction처럼 특정 액션에만 필요한 값은 Route State로 전달한다. Session/Settings State 키는 전역값을 우선 적용해 오래된 Route State가 로그인·권한·구독·설정 완료 상태를 다시 Guest/권한 전/무료/미설정 상태로 되돌리지 않게 한다. 이 기준으로 A2/A3, O3, G6를 거친 뒤 하단 탭이나 프리뷰 네비로 이동해도 accountLinked, permissionReady, locationReady, premiumActive 상태가 끊기지 않게 검증한다. A3 일반 완료는 accountLinked를 남기고 H1로 복귀하며, O3 일반 완료는 locationReady/permissionReady를 남기고 O4로 이동한다. G6 일반 구독 완료는 premiumActive/accountLinked를 남기고 G5로 이동해 이후 MY·여행·코스 화면에서 같은 구독 상태를 사용한다. 위치명 탭은 H2 위치 변경으로, 벨 아이콘은 H3 알림 사이드바로 이동한다. H2~H5에는 LOCATION/NOTIFICATION/UMBRELLA/RAIN STATE를 두어 위치 선택, 알림 딥링크, 우산 추천, 강수 타임라인의 현재 상태를 검증한다. NOTIFICATION STATE는 H1 카드 노출 상태와 별개로 유지해, 홈에서 확인 완료로 사라진 우산 추천도 H3에서 다시 열 수 있게 한다. H2의 현재 위치 사용과 검색 위치 조회는 Guest도 가능하지만, 새 위치 저장은 A2 계정 연결 후 적용한다. Guest가 새 위치 저장을 누르면 A2/A3 완료 후 returnTo=H2, 선택 위치/검색어, resumeLocationSave 상태로 복귀해 위치 저장 완료 피드백을 보여준다. AI 장소 추천 카드는 P3 목적지 필터/상세 허브로 직접 연결한다. 저장·동기화·Weather Note 작성처럼 계정 가치가 필요한 액션은 A2 계정 연결 화면으로 이동시키고, A2에서 추천 로그인 수단과 Guest 유지 가능성을 안내한다.
MY 연결 원칙: MY 탭은 M1 허브로 진입하고, M1에서 알림 설정(M2), 전역 설정(M3), 계정 관리(A4), 스타일 기준(O4), 프리미엄(G6)으로 분기한다. M1은 accountLinked가 없으면 Guest 모드와 계정 연결 CTA를 보여주고, 프리미엄 업그레이드·프로필 편집처럼 계정이 필요한 액션은 A2로 먼저 보낸다. A4는 계정 전용 화면으로 유지해 연결 계정, 구독, 데이터 내보내기, 로그아웃/탈퇴만 다루고 알림 설정(M2)·스타일 기준(O4)은 M1 허브에서만 진입한다. 프리미엄 업그레이드에서 A2/A3를 거친 경우에는 단순히 M1로 돌아오지 않고 G6 페이월을 자동으로 열어 원래 사용자의 업그레이드 의도를 계속 처리한다. M1/A4에서 구독 관리로 진입한 경우 G6 완료 후 returnTo=M1/A4로 복귀해 premiumActive 상태를 즉시 표시한다. M3의 위치 관리와 알림 권한 관리는 H2/O3로 이동하되 returnTo=M3를 유지해 완료 후 GLOBAL STATE에 위치 관리 확인/권한 정상 상태를 즉시 반영한다. M1/M3/A4에는 PROFILE/GLOBAL/ACCOUNT STATE를 두어 계정 연결, 단위·권한, 구독 활성, 데이터 내보내기·탈퇴 확인 상태를 검증한다. A4의 로그아웃과 회원 탈퇴 확정은 accountLinked/premiumActive를 Session State에서 해제하고 H1 Guest 홈으로 복귀해, 이후 코디 저장·목적지 알림·프리미엄 기능이 다시 계정 gate를 타도록 한다. 이때 목업 프리뷰의 Saved Destination List와 Destination Care State도 함께 비워 계정 기반 데이터가 Guest 상태 화면에 남지 않게 한다. H4/H5/G2처럼 알림 설정이 목적지인 CTA는 M1을 거치지 않고 M2로 바로 이동한다. 이때 M2는 originReturnTo와 alertFocus를 받아 우산·비·목적지 알림 중 어떤 설정을 조정하는지 바로 알 수 있게 표시하고, 뒤로가기는 M1이 아니라 원래 화면으로 복귀한다. M2의 시스템 알림 권한 상태는 Session State의 permissionReady를 기본값으로 사용해, O3를 거치지 않은 사용자는 "알림 권한 확인 필요" 상태로 시작하고 O3 완료 후에만 정상 상태로 복귀한다. M2의 스마트 알림 ON/OFF, 고급 설정 열림 상태, 강수 상세·출근 외출·자기 전 체크·목적지 여행·방해 줄이기 토글은 Settings State에 남겨 다른 화면으로 이동해도 사용자가 조정한 상태가 유지되게 한다. 알림 설정 요약으로 알아서 챙기기 ON/OFF, 고급 설정 열림/닫힘, 세부 토글 ON 개수를 표시해 목업에서 설정 유지 여부를 직접 검증한다.
O2~O6 온보딩 연결 원칙: O1은 선택 온보딩 진입용 브랜드 모먼트로 사용하며, 목업에서는 O2 온보딩 시작과 H1 홈 복귀 분기를 함께 검증한다. O2의 기능 소개 건너뛰기와 추천 기준 설정 CTA는 모두 O3 권한 요청으로 이동한다. O3는 권한 허용 여부와 관계없이 O4로 진행 가능하며, 권한 상태 카드는 0/2~2/2와 거부 상태, 필요한 알림 반영 범위를 표시한다. 거부 상태에서도 검색 위치와 앱 내 안내는 유지하고, 현재 위치 자동화와 푸시만 제한한다. 거부 상태는 MY 전역 설정(M3) > 권한 관리(Z4)에서 재요청한다. O4는 온보딩 진입 시 스타일·핏·연령대 저장 후 O5 스마트 알림 기준으로 이어지며, 선택 태그 수·핏·연령대·준비/선택 필요 상태를 보여준다. M1/C1에서 스타일 기준 수정으로 진입한 경우에는 설정 모드로 동작하고 저장 후 returnTo로 복귀해 styleProfileSaved/selectedStyles/fit/age를 Settings State에 남긴다. O5는 사용 상황·필수 날씨 ON·목적지 확장 예정 상태를 보여준 뒤 O6 목적지 등록으로 이어진다. O6에서 목적지를 등록하면 G1 출발 메인으로 이동해 목적지 알림 기준을 세밀하게 맞추고, 나중에 하기를 선택하면 H1 Guest 홈으로 복귀한다.
A1/A2/A3 계정 연결 원칙: A1은 가입 선택 없이 H1 Guest 홈으로 진입하는 브랜드 모먼트이며, 목업에서는 홈 이동 액션으로 자동 진입 의도를 검증한다. A2는 저장·동기화·프리미엄처럼 계정 가치가 있는 액션에서만 호출하고, 로그인 수단을 선택하면 A3 약관 동의로 이동한다. A2 실제 앱 화면에는 계정 연결 이유, 자동 추천 로그인 1~2개, 다른 방법 보기, 나중에 하기, 약관 안내만 노출한다. 계정 연결/코디 저장/옷장 저장/장소 저장/알림 추가/여행 저장/프리미엄 진입/Weather Note 작성/날씨 제보 저장 중 어떤 액션에서 호출되었는지, 저장 대기/약관 필요/Guest 유지 상태 같은 검증용 정보는 Route State와 문서에만 유지하고 실제 앱 UI에는 노출하지 않는다. A2에서 나중에를 선택하거나 뒤로가면 returnTo가 있는 계정 gate 진입은 원래 화면 상태로 복귀하고, 일반 진입은 H1 Guest 홈으로 복귀한다. A3에는 약관 상태 카드를 두어 필수 완료/대기, 선택 ON/OFF, 원래 액션 복귀 예정 상태를 표시하고 CTA에도 "동의하고 [원래 액션] 계속" 맥락을 반영한다. A3는 returnTo가 없는 일반 진입에서는 H1로 복귀하고, A2가 returnTo와 선택 데이터를 전달한 경우에는 사용자가 시도했던 저장·알림·프리미엄 액션으로 되돌아간다.

08 · Notification System
알림 체계 전체 구조
Smart Notification Principle
WeatherON 알림은 사용자가 시간표를 일일이 만드는 구조가 아니라 주 사용 상황 1개만 고르면 시스템이 상황에 맞춰 챙기는 알아서 챙기기로 설계한다. 강수 임박, 폭염·한파·강풍 특보처럼 행동이 필요한 날씨 알림은 필수 축으로 유지하고, 출근·외출·여행·자기 전 내일 체크는 사용자의 선택 기준과 목적지/일정/날씨 변화에 따라 자동 보정한다. 알림 설정 화면(M2)의 기본 화면은 스마트 알림 ON/OFF + 적용 중인 알림 3개 요약 + 고급 설정 진입만 제공하고, 세부 토글과 시간 조정은 고급 설정에 숨긴다. H4/H5/G2에서 M2로 진입하는 경우에는 returnTo=H4/H5/G2, alertFocus=umbrella/rain/destination을 유지해 설정 완료 후 원래 맥락으로 돌아간다. 스마트 알림을 끄면 생활 루틴·목적지 알림은 중지하되 긴급성 높은 필수 날씨만 유지하며, 시스템 알림 권한이 꺼진 경우는 별도 상태 카드와 권한 관리 진입으로 복구 흐름을 제공한다. M2에서 권한 관리를 누르면 O3 완료 후 returnTo=M2, resumePermissionSettings=true로 복귀해 사용자가 보던 알림 설정 맥락을 유지한다. M2에서 바꾼 스마트 알림과 고급 설정 토글은 Settings State로 저장해 H4/H5/G2 바로가기나 MY 탭 이동 후에도 상태가 초기화되지 않게 한다. 이때 originReturnTo도 함께 보존해 O3를 거쳐도 H4/H5/G2 원래 화면 복귀가 끊기지 않게 한다. 목적지·여행 알림은 Guest 상태에서 조회와 준비 가이드 확인은 허용하되, 저장 목적지 기반 발송은 계정 연결과 알림 권한이 모두 준비된 뒤 적용한다. M2에서 목적지 알림 계정 연결을 누르면 A2/A3 완료 후 returnTo=M2, resumeAlertSettings=true로 복귀해 사용자가 보던 알림 설정 맥락을 유지한다. 이 구조의 목표는 사용자가 시간을 매번 직접 정하는 것이 아니라, 처음 2~3번의 기준 설정 이후 WeatherON이 날씨·목적지·사용 패턴에 맞춰 자동 보정하는 것이다.

P3 목적지 허브 원칙: P3는 단순 필터 리스트가 아니라 저장 목적지의 상태 허브로 둔다. 리스트 상태에는 알림 상태 카드를 두어 현재 필터의 목적지 수와 목적지 추가 CTA를 제공한다. 필터 결과가 없을 때는 빈 카테고리 상태를 보여주고 P1 목적지 추가 또는 전체 보기로 바로 복귀할 수 있게 한다. 사용자가 목적지 카드를 탭하면 같은 화면 안에서 상세 상태로 전환되어 선택 목적지 컨텍스트, 상세 상태, 출발지/목적지 날씨 비교, 장소별 준비 추천, 목적지 알림 ON/OFF를 확인한다. 상세 상태는 선택 목적지명/카테고리/다음 화면 전달 상태와 계정 필요/권한 필요/준비/켜짐 상태를 함께 사용해 계정 연결 전, 알림 권한 전, 켜기 가능, 적용 완료를 구분한다. G1 출발 메인은 저장 목적지 운영의 기본 진입점으로 두고, 출발 상태에서 목적지 수, 알림 ON 수, 날씨 비교·출발·신발·우산 알림 적용 범위를 보여준다. 목적지 카드는 G2 목적지 알림 상세로 이동한다. G1의 검색창·추가 버튼은 P1 목적지 추가로 연결하며, P1에서 목적지 알림에 추가하면 Saved Destination List에 저장하고 완료 피드백 후 G1 목록으로 복귀한다. P1에는 추가 상태 카드를 두어 검색 결과 수, 선택된 목적지, Guest/계정 저장 상태, 저장 완료 상태를 표시한다. 검색 결과가 없을 때는 전체 보기와 재검색 흐름을 제공하고, 목적지가 선택되지 않은 상태에서는 목적지 알림 추가 CTA를 비활성화한다. Guest는 목적지 조회와 미리보기까지 가능하지만 영구 저장과 목적지 알림 반영은 A2 계정 연결 후 진행한다. P1에서 Guest가 목적지 저장을 탭하면 A2에 returnTo=P1, pendingAction=장소 저장, 선택 목적지, resumeSave 상태를 전달하고, A3 약관 완료 후 P1로 복귀해 원래 저장 액션을 자동 완료한다. 계정 연결 상태에서 P1 저장을 완료하면 해당 목적지는 G1 출발 목록과 P3 목적지 필터 목록에 동시에 반영되고, Destination Care State는 ON으로 저장된다. P3 상세에서 목적지 알림을 켤 때 계정 미연결이면 A2, 알림 권한 전이면 O3 권한 단계로 이동시켜 저장과 발송 조건을 명확히 한다. 이때 selectedDestination, resumeCare, accountLinked/permissionReady 상태를 함께 전달해 A2/A3 또는 O3 완료 후 같은 P3 상세로 돌아오고, 조건이 모두 충족되면 목적지 알림 상태를 ON으로 이어간다. P3 상세 상태에서는 선택 목적지 컨텍스트를 유지한 채 카테고리별 준비 가이드(P2)와 출발 알림 상세(G2)를 모두 연결해 장소 콘텐츠와 알림 운영이 분리되지 않게 한다. 목업 프리뷰 라우터도 P3에서 선택한 목적지 데이터와 accountLinked/permissionReady/careOn 상태를 P2/G2로 전달해, 잠실·북한산·해운대 중 어떤 카드를 눌러도 다음 화면의 제목·날씨 비교·추천 문구가 같은 목적지 기준으로 이어지게 한다. P2에서는 GUIDE STATE로 등록 목적지 기반 준비 가이드임을 표시하고, 목적지 알림 보기로 G2에 진입하며, 다른 목적지 보기/뒤로로 P3 필터 목록에 복귀한다. P2에서 G2로 이동할 때는 selectedDestination을 유지해 준비 가이드에서 알림 상세로 이어지는 흐름이 같은 목적지를 바라보게 한다. 뒤로가기는 필터 리스트 상태로 복귀하며, 상세 상태의 목적지 알림 설정은 G1/G2의 목적지 알림과 동일 데이터로 동기화한다. 목업 프리뷰에서는 공유 Saved Destination List와 Destination Care State를 두어 P1에서 추가한 목적지가 G1/P3에 동시에 보이고, G2 또는 P3에서 특정 목적지 알림을 켜면 G1 카드의 준비/켜짐 배지, G2 알림 상태, P3 상세 상태가 같은 값으로 갱신되게 한다. G2에는 알림 상태 카드를 두어 계정 필요/권한 필요/준비/켜짐 상태와 다음 알림 적용 여부를 보여주고, 켜기 CTA는 계정 미연결이면 A2/A3, 알림 권한 전이면 O3를 거친 뒤 같은 G2로 돌아오게 한다. G2는 공유 careState뿐 아니라 M2/O3 복귀 routeState.careOn·resumeCare도 함께 확인하고, 조건이 충족되면 공유 careState로 다시 동기화해 권한/계정/알림 설정을 거친 뒤에도 목적지 알림 ON 상태가 끊기지 않게 한다.
여행 플래너·프리미엄 연결 원칙: G1의 여행 플래너 배너는 G3 여행 플래너로 연결하고, G3는 TRIP STATE에서 무료 상태, 프리미엄 활성, 공유 준비, 미구독 시 G6 페이월 연결 상태를 보여준다. 무료 상태에서 일정 추가·여정 공유를 탭하면 G6로 이동하고, 구독 완료 후 returnTo=G3로 복귀해 일정 추가는 G4로 계속 진행하고 여정 공유는 공유 준비 완료 상태로 이어진다. 프리미엄 활성 상태의 일정 추가는 무료 도보여행 코스 날씨(G4)로 이어진다. G4는 코스 날씨 조회는 무료로 제공하되, 코스 저장과 출발 전 알림은 계정 연결 + 알림 권한이 준비된 뒤 적용한다. Guest 코스 저장은 A2/A3 완료 후 returnTo=G4와 selectedCourse를 유지해 돌아오고, 알림 권한 전 상태는 O3 완료 후 같은 코스 상태로 복귀한다. 코스 알림 상태는 계정 필요/권한 필요/준비/한도 초과/프리미엄/프리미엄+권한 필요/켜짐 상태로 계정 미연결, 권한 전, 등록 가능, 무료 2개 한도 초과, 프리미엄 활성, 프리미엄 활성+권한 전, 알림 등록 완료를 구분한다. 장거리 국토종주나 AI 분석이 필요한 경우 G6 페이월을 거쳐 G5 국토종주 AI 플래너로 진입한다. G6는 구독 상태에서 미구독/구독 완료를 표시하고, 일반 진입의 구독 완료 CTA는 G5로 이어진다. G4 무료 한도 초과처럼 returnTo가 있는 진입은 premiumActive=true와 원래 선택 데이터를 유지해 원래 저장/알림 액션으로 복귀한다. G5는 여정 점검 상태로 프리미엄 활성, 알림 권한, 저장/공유 상태를 명시한다. 예보 업데이트 재분석 알림과 AI 플랜 저장은 알림 권한이 필요하며, 권한 전 상태에서 O3 권한 단계로 연결한다. O3 완료 후 returnTo=G5로 복귀해 출발지/목적지/출발일/일수와 저장 의도를 유지하고, 저장 CTA에서 진입한 경우 플랜 저장 완료 상태로 이어진다. AI 플랜 저장 후 저장됨, 여정 공유 후 공유 준비 상태로 전환해 프리미엄 플래너의 결과 보존과 공유 준비 상태를 구분한다.

11-3 · Community Weather Report
커뮤니티 날씨 제보

기상청 관측소는 지점 기반이다. 골목까지 못 잡는다. 티맵·웨이즈가 사고를 실시간으로 잡는 것처럼, WeatherON 사용자가 현재 위치의 실제 날씨를 탭 한 번으로 제보한다. 제보가 쌓이면 공식 예보보다 빠르고 촘촘한 로컬 날씨 레이어가 만들어지고, 이 데이터는 WeatherON만의 독자 자산이 된다. v2.0 출시 · MAU 5만 달성 후 활성화.

CWR FLOW STATE. W1은 H1 홈의 보조 참여 흐름으로 주변 제보 카드와 작은 제보 CTA를 제공하고, W2는 날씨 선택과 GPS 자동 제보만 담당한다. 위치 권한 전 상태에서 제보하기를 탭하면 pendingAction=날씨 제보 저장, returnTo=W2로 O3 권한 단계에 연결하고, O3 완료 후 selectedWeather, resumeWeatherReport 상태로 돌아와 선택 날씨와 GPS 제보 준비 상태를 유지한다. Guest 제보는 익명 처리 가능하되 배지/이력 저장은 A2 계정 연결 후 반영한다. 배지/이력 저장을 누르면 A2/A3 완료 후 returnTo=W2, resumeReportAccount 상태로 복귀해 계정 제보 상태를 표시한다. W3는 신뢰도 결과와 배지 갱신을 즉시 보여주며, W4는 MY에서 제보 기록·확정률·지역 랭킹으로 재진입한다. 각 화면은 CWR STATE, REPORT STATE, BADGE STATE, HISTORY STATE를 노출해 미확정/선택/확정/이력 상태를 검증한다.

Value 1 · 현장성
관측소 사각지대 해소
기상청 관측소 간격 평균 5~15km. 사용자 제보가 500m 단위 실시간 레이어를 만든다. "내 골목은 지금 비 안 와요" 같은 초로컬 정보.

F1 · 탭 한 번 날씨 제보

14 · AdMob Monetization
구글 AdMob 광고 수익화 — 심사 요건 완전 정리

⚠️ 중요 — 모바일 앱은 애드센스가 아닌 AdMob. 애드센스(AdSense)는 웹사이트 전용이고, 앱 광고 수익화는 Google AdMob이 공식 채널이다. AdMob 게시자도 Google 애드센스 프로그램 정책을 함께 준수해야 하므로, 아래 요건은 두 정책을 모두 반영했다. 모든 신규 앱은 app-ads.txt 인증 + 앱 준비 상태 검토를 통과해야 광고가 완전히 게재된다.

ADMOB FLOW STATE. R1은 M3 전역 설정의 정책 허브로 진입하고 개인정보처리방침(R2), 광고 동의 관리(R3), 이용약관(R5), 위치기반서비스 이용약관(R6), 오픈소스 라이선스(R7), 계정/알림 설정으로 분기한다. R2/R3/R4/R5/R6/R7에는 각각 개인정보 문서, 광고 동의 상태, 광고 배치 상태, 서비스 약관, 위치 약관, 라이선스 고지를 두어 앱 내 열람성과 심사 대응성을 검증한다. R4 홈 광고는 AI 추천 카드 아래 1개만 배치하고 버튼 인접 배치를 피하며 AD 라벨과 구분선을 유지한다.

A2 로그인 화면은 앱 첫 관문으로 노출하지 않는다. 사용자가 계정 연결, 코디 저장, 옷장 저장, 장소 저장, 알림 추가, 여행 저장, 프리미엄 진입, Weather Note 작성, 날씨 제보 저장처럼 계정이 필요한 기능을 탭했을 때 호출한다. 실제 앱 UI는 계정 연결 이유, 자동 추천 로그인 1~2개, 다른 방법 보기, 나중에 하기, 약관 안내만 노출하는 경량 구조로 유지한다. UI 언어는 기기 언어/사용자 설정을 유지하고, 로그인 방법은 계정 환경 기반으로 자동 추천한다. 우선순위는 스토어 국가, SIM/전화 국가, OS 지역, 기기 언어, 타임존, 현재 위치 순서이며 현재 위치는 여행·출장 가능성이 높으므로 마지막 보조 신호로만 사용한다. 국가/권역 자동추천은 유지하되, 국가/지역 선택 또는 변경 버튼은 계정 연결 화면에 노출하지 않는다. "다른 방법 보기"는 지역 변경이 아니라 현재 자동 추천 권역 안의 추가 로그인 수단을 펼치는 용도다. A3 약관 동의 후 위치/알림 권한이 비어 있으면 O3 앱 준비 화면에서 통합 권한 안내를 거치고, 권한 허용 또는 나중에 설정 후 호출 액션으로 복귀한다. 지역별 우선순위, 특수 환경 국가 제외, 호출 액션·복귀 상태 등 검증용 정보는 본 기획서와 플로우차트에만 유지한다.

19 · ON Square NEW
소셜 레이어 — 날씨를 함께 경험하는 공간

날씨가 같은 곳에 있는 사람들은 같은 세계에 살고 있다. ON Square는 날씨 앱 위에 얹는 소셜 레이어의 공간 이름이다. 동일 도시 또는 여행지를 설정한 유저들이 하나의 날씨 공간 "ON Square"에 공존하고, 실제 날씨 데이터가 그 세계의 물리 법칙이 된다. 유틸리티 앱을 넘어 "날씨를 함께 경험하는 공간"으로 확장하는 장기 전략 기능. 앱 런칭과 함께 ON Square Lite를 공개하고, 운영 안정화 후 기능을 단계 확장.

SOCIAL FLOW STATE
소셜 탭은 계정 연결 화면이 아니라 ON Square 진입점이다. 최초 유저는 S0에서 대표 마스코트를 받고 S1로 이동하며, 기존 유저는 S1로 바로 진입한다. S1은 단순 캐릭터 대시보드가 아니라 "오늘의 날씨 미션"을 먼저 보여주는 광장 허브다. 사용자는 체크인·제보·공감 중 하나를 1탭으로 수행하고, 결과가 컴패니언 반응, XP/포인트 피드백, 동네 도움 지표에 즉시 반영된다. 상태 표기는 ON SQUARE STATE, SQUARE STATE, NOTE STATE, REACTION STATE로 나눠 최초 진입/체크인/핀 작성/공감 완료/보상 반영을 검증한다. Weather Note는 Guest 읽기를 허용하고, 핀 작성은 A2 계정 연결 gate, 내 위치 제보는 O3 권한 gate로 분리한다. Guest가 핀 작성을 누르면 A2/A3 완료 후 returnTo=S2, resumeNoteWrite 상태로 복귀해 작성 준비 완료 상태를 보여주고, 내 위치 제보는 O3 완료 후 returnTo=S2, resumeLocationReport 상태로 돌아와 위치 제보 준비 상태를 유지한다. Guest는 읽기·탐색 가능, 작성·저장·보상성 액션은 A2 계정 gate로 분리한다. MVP에서는 완전한 경제 시스템이 아니라 즉시 피드백용 포인트·배지 조각만 표현하고, 상점·교환 구조는 출시 후 확장한다.

앱 상주시간 & 방문수 증대 전략

날씨 앱의 구조적 한계: 정보 소비가 5초 안에 끝난다. 기존 날씨 앱의 평균 세션 시간은 30초 내외, 방문 패턴은 "아침 1회 확인 후 이탈"이 대부분이다. WeatherON의 궁극적 목표는 이 구조를 깨는 것이다 — 날씨를 확인하러 들어왔다가 머물고 싶어지는 앱. 방문수와 상주시간은 광고·구독·제휴 수익 모두의 기반이며, MAU 성장 속도를 결정하는 핵심 지표다.

3순위 — 선택 참여: 희귀 기상 이벤트, 제보 반영 피드백, Weather Mail 수신.

알림 피로를 막기 위해 하루 최대 3건으로 제한. 기본은 알아서 챙기기이며, M2 기본 화면은 스마트 알림 ON/OFF, 권한/발송 상태 태그, 권한 관리·테스트 알림 빠른 액션, 필수 날씨·생활 루틴·목적지 출발 3행만 제공한다. 세부 on/off, 시간 조정, 피로도 제어는 고급 설정에 숨긴다.

UI 디자인 컨셉 — Liquid Glass

WeatherON은 Apple의 Liquid Glass 디자인 언어를 채택한다. 날씨 앱의 특성상 자연의 유동성과 빛, 투명감을 UI에 그대로 녹여내는 방식이 브랜드 아이덴티티와 가장 높은 정합성을 보인다. 불투명 솔리드 카드 대신 반투명 유리층 + 백드롭 블러로 날씨 배경이 카드 너머로 비치며, 시간·날씨 상태에 따라 배경 그라디언트가 변화할 때 UI 전체가 함께 살아 숨쉰다.

핵심 디자인 원칙
🪟
투명 유리층 (Glass Layer)
모든 카드·모달·시트는 background: rgba(255,255,255,0.08~0.14) + backdrop-filter: blur(20px) 조합. 솔리드 흰 배경 사용 금지.
🌈
날씨 반응형 배경
맑음·흐림·비·눈·밤에 따라 배경 그라디언트 변경. 유리층 너머로 배경색이 비치며 UI 전체 분위기가 자동으로 전환됨.
💧
유동성 (Fluid Motion)
카드 진입·전환 애니메이션은 스프링 기반 easing. 탭바·모달 전환 시 유리 표면이 '흘러오는' 느낌. iOS 26 네이티브 애니메이션 API 활용.
✨
내부 반사 (Specular Highlight)
유리 카드 상단에 linear-gradient(rgba(255,255,255,0.18), transparent) 1px 하이라이트 레이어 추가. 빛 굴절 효과 재현.

날씨 다이나믹 배경 애니메이션 — 홈(H1) NEW

위 6종 그라디언트는 정적 색상값이다. 홈 화면(H1)에서는 한 단계 더 나아가 현재 위치 날씨에 맞춘 미동(微動) 파티클 레이어를 그라디언트 위에 추가해, "정보를 더 보여주는" 것이 아니라 "지금 날씨를 직관적으로 체감"하게 만든다 — Quiet Horizon 철학(결정을 더 빠르게 만드는가)과 동일한 검증 기준을 따른다. 애니메이션은 항상 z-index 최하단(배경)에 위치하며 카드·CTA·텍스트의 대비를 해치지 않는다.

• 라이트 모드에서도 동일한 모티프를 유지하되 opacity·색상만 라이트 토큰으로 리틴트(예: 빗줄기는 SKY 단색 대신 더 어두운 톤으로 낮춤)
• 성능: 60fps 목표, 파티클 수는 디바이스 등급별로 분기(저사양 기기 50% 감축) — 기존 "투명 효과 줄이기"와 별개로 "배경 애니메이션 줄이기" 옵션 추가 고려
• 화면이 백그라운드/잠금 상태일 때는 애니메이션 정지(배터리 보호)
• 확장 아이디어: 일출·일몰 실제 시각(날씨 상세 화면에서 이미 계산 중인 데이터 재사용) 기준으로 CLEAR-DAY ↔ NIGHT 그라디언트를 크로스페이드 전환
