# WeatherON 기능 출시 로드맵

> 기준일: 2026-07-09
> 목적: 기획안 피드백 반영 후 개발/출시 범위를 줄이고, 실제 사용 검증 순서대로 기능을 배치한다.

---

## 1. 결론

WeatherON은 전체 기능을 한 번에 출시하지 않는다.

1차 출시는 "예쁜 종합 날씨 앱"이 아니라 사용자가 실제로 반복 사용하는지 확인하는 검증용 MVP로 잡는다.
핵심 검증 대상은 아래 항목이다.

- 출발시간 역산
- 비 시작/그침 타임라인과 알림 신뢰성
- 날씨 기반 기본 코디 추천과 저장/옷장 진입 흐름
- 우산·겉옷·신발처럼 바로 챙길 것 판단

코디 기능은 정식 출시 단계에 포함한다.
현재 진행 중인 Android 비공개 테스트도 14일 운영 안에 코디 추천, 코디 상세, 저장 gate, 옷장 프리셋 진입을 포함한 빌드로 전환한다.
우산, 신발 추천은 코디와 함께 홈 경험을 돕는 준비 판단으로 유지한다.
소셜, 캐릭터, CP, 커뮤니티 제보, 여행 플래너, AI 장소 추천, 광고, 구독 결제는 MVP 이후로 둔다.

UI 디자인과 컴포넌트 기준은 기능 출시 범위와 별개로 `docs/design/WeatherON_UI_Design_Spec.md`를 따른다.
목업과 와이어프레임이 다르면 목업 기준을 우선한다.
`docs/Project Wind/`는 출시 후 대규모 UI 업데이트를 위한 신규 디자인 시스템 연구 트랙으로 분리한다.

---

## 2. 왜 조정하는가

기획안 피드백의 핵심은 기능 범위가 너무 크다는 점이다.
현재 기획에는 날씨 코디, 신발/우산 추천, 출발시간 역산, 소셜, 캐릭터, CP 경제, 커뮤니티 제보, 도보여행, 해외여행, AI 추천, 광고, 구독 결제가 함께 들어 있다.

이 범위를 한 번에 개발하면 출시 전 시간이 길어지고, 핵심 기능의 실제 사용성 검증이 늦어진다.
따라서 기능 출시 순서는 "핵심 사용 이유 검증 -> 안정화 -> 확장"으로 바꾼다.

## 3-1. 현재 기능 검증 우선순위

2026-07-08 현재 기능 검증은 `홈/코디/출발/MY` 흐름을 우선한다.
단, 탭바 구성, 아이콘, 카드 반경, CTA, 상태 톤 같은 UI 스펙은 `docs/design/WeatherON_UI_Design_Spec.md`를 따른다.
MVP 범위에서도 디자인 스펙을 낮추지 않는다.

| 우선 흐름 | 현재 화면 | 검증 역할 |
|---|---|---|
| 홈 | H1 | 날씨 히어로, 목적지/강수 quick-action, 오늘 준비 요약 |
| 코디 | C1/C2/C3/C4 | 날씨 근거형 코디 추천, 옷장 프리셋, 저장 gate, 코디 상세 |
| 출발 | G1 | 목적지 목록, 목적지 추가, 목적지 없음 empty state |
| MY | M1 | 계정, 준비 상태, 권한, 알림, 표시, 정책 진입 |

`W1~W4`, `G3~G6`, `P2/P3`, `R3/R4`, `S0~S3`는 기능 출시 판단을 보류할 수 있지만, 아이콘/컴포넌트/상태 표현은 통합 디자인 스펙을 따른다.
확장 라우트는 실제 사용자 검증 지표가 나온 뒤 단계별로 노출 여부를 다시 판단한다.

## 4. MVP 0 범위

MVP 0는 내부 검증용이다.
목표는 "앱이 무엇을 해주는지 5초 안에 이해되는가"와 "나갈 시간/비 그침/챙길 것 판단이 유용해 보이는가"를 확인하는 것이다.

### 포함

- O2 온보딩: 출발시간 역산, 목적지 날씨 비교, 비 그침 알림 중심 소개
- H1 홈: 나갈 시간/비 그침/챙길 것 시각 카드, 목적지/강수 quick-action, 오늘 준비 요약
- G1/G2: 목적지 목록, 목적지 없음 empty state, 출발 준비 판단
- H5: 강수 타임라인과 비 그침 알림 토글
- H4/C1/C2/C3/C4: 우산 추천, 기본 코디 추천, 옷장 프리셋, 코디 상세
- A2/A3/A4: 저장/알림 확장 전 계정 gate
- M1/M2/M3/M4/R1~R4: MY 허브, 권한, 설정, 정책 접근
- UI 스펙: `docs/design/WeatherON_UI_Design_Spec.md` 기준 적용

### 제외

- ON Square 정식 소셜 루프
- Weather Note/Reaction/CP/아이템샵
- W1~W4 커뮤니티 날씨 제보
- G3/G4/G5 해외/도보/AI 여행 플래너
- G6 구독 결제
- AdMob 실제 광고 송출
- 소셜/여행/광고 기능 정식 활성화
- 코디 개인화 고도화와 AI 대화형 재추천

---

## 5. MVP 1 범위

MVP 1은 소수 사용자에게 배포해 실제 반복 사용 여부를 확인한다.
UI 추가보다 알림 신뢰성, 목적지 저장, 권한 복구를 우선한다.

### 포함

- 목적지별 출발 권장 시간 저장
- 비 시작/그침 알림 조건 저장
- 푸시 알림 수신, 열람, 딥링크
- 알림 실패/권한 거부/fallback 상태 문구
- C1/C4 기본 코디 추천과 상세 보기
- C2/C3 옷장 프리셋 및 저장 gate
- 테스트 사용자 피드백 기록 양식: `docs/planning/WeatherON_MVP1_테스트_피드백_기록_양식.md`

### 측정

- 첫 실행 후 핵심 기능 이해 여부
- 출발 알림이 실제 외출 판단에 도움이 되는지
- 비 그침 알림을 다시 켜고 싶은지
- 알림이 제때 오지 않았을 때 이탈하는지
- 코디 추천을 날씨 판단의 일부로 이해하는지
- 코디/우산 보조 기능이 핵심 판단을 방해하지 않는지

## 6. 확장 보류 기준

아래 기능은 MVP 사용 지표가 나오기 전까지 개발 우선순위를 낮춘다.

| 기능 | 보류 이유 | 재개 조건 |
|---|---|---|
| 소셜/ON Square | 핵심 날씨 판단과 별도 루프 | MVP 재방문 지표가 충분하고 커뮤니티 필요성이 확인될 때 |
| CP 경제/캐릭터 시스템 | 운영/밸런스 비용 큼 | 리텐션 개선 실험으로 필요성이 확인될 때 |
| 도보/해외여행 플래너 | 데이터/API/일정 UX 범위 큼 | 목적지 날씨 비교 사용률이 높을 때 |
| AI 장소 추천 | 비용과 품질 검증 필요 | 룰 기반 장소 추천으로 수요 확인 후 |
| 광고/구독 결제 | 초기 사용성 검증을 흐림 | MVP 핵심 기능 재방문이 확인된 뒤 |
| 코디 AI 고도화 | 기본 룰엔진 검증 후 품질 투자 필요 | 비공개 테스트에서 코디 이해도와 재방문 지표가 기준 충족할 때 |
