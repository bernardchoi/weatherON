# WeatherON 기능 출시 로드맵

> 기준일: 2026-06-29
> 목적: 기획안 피드백 반영 후 개발/출시 범위를 줄이고, 실제 사용 검증 순서대로 기능을 배치한다.

---

## 1. 결론

WeatherON은 전체 기능을 한 번에 출시하지 않는다.

1차 출시는 "예쁜 종합 날씨 앱"이 아니라 사용자가 실제로 반복 사용하는지 확인하는 검증용 MVP로 잡는다.
핵심 검증 대상은 아래 3개다.

- 출발시간 역산
- 목적지 날씨 비교
- 비 시작/그침 타임라인과 알림 신뢰성

코디, 우산, 신발 추천은 홈 경험을 돕는 보조 기능으로 유지하되, 개발 우선순위는 출발/강수 판단보다 낮춘다.
소셜, 캐릭터, CP, 커뮤니티 제보, 여행 플래너, AI 장소 추천, 광고, 구독 결제는 MVP 이후로 둔다.

---

## 2. 왜 조정하는가

기획안 피드백의 핵심은 기능 범위가 너무 크다는 점이다.
현재 기획에는 날씨 코디, 신발/우산 추천, 출발시간 역산, 소셜, 캐릭터, CP 경제, 커뮤니티 제보, 도보여행, 해외여행, AI 추천, 광고, 구독 결제가 함께 들어 있다.

이 범위를 한 번에 개발하면 출시 전 시간이 길어지고, 핵심 기능의 실제 사용성 검증이 늦어진다.
따라서 기능 출시 순서는 "핵심 사용 이유 검증 -> 안정화 -> 확장"으로 바꾼다.

---

## 3. 출시 단계

| 단계 | 목표 | 포함 기능 | 제외/보류 | 통과 기준 |
|---|---|---|---|---|
| MVP 0 | 내부 프로토타입 | 온보딩, 홈, 이동수단 선택/도착 희망 직접 입력 기반 출발시간 역산 UI, 목적지 날씨 비교 UI, 강수 타임라인, 기본 코디/우산 보조 카드 | 실제 결제, 소셜, CP, 여행/AI 플래너 | `npm run check:android-product-quality`, `npm run build:mockups`, web export 통과 |
| MVP 1 | 소수 사용자 검증 | 목적지 저장, 출발 알림 조건, 비 시작/그침 알림, 권한/계정 gate, 알림 이력 | 광고, 구독, 커뮤니티, ON Square 확장 | 10~30명 테스트에서 알림 정시성/재방문 피드백 확보 |
| Beta | Android 폐쇄 테스트 | 실기기 안정화, 푸시 알림 서버 경로, 스토어 스크린샷, 개인정보/약관, 오류/fallback UX | 대형 기능 추가 금지 | Play 폐쇄 테스트 운영 요건 충족, 주요 blocker 해소 |
| v1.0 | 공개 출시 후보 | MVP 1 기능 안정화, 최소 정책/스토어 자산, 알림 신뢰성 모니터링 | 새 수익화 기능 금지 | 핵심 알림 실패율과 주요 UX 이슈 허용 기준 이내 |
| v1.1+ | 검증 후 확장 | 코디/신발 고도화, 도보여행, AI 장소 추천 일부 | 소셜/CP/구독은 반응 확인 후 별도 결정 | MVP 사용 지표가 확장 기준 충족 |

---

## 4. MVP 0 범위

MVP 0는 내부 검증용이다.
목표는 "앱이 무엇을 해주는지 5초 안에 이해되는가"와 "출발/강수 판단이 유용해 보이는가"를 확인하는 것이다.

### 포함

- O2 온보딩: 출발시간 역산, 목적지 날씨 비교, 비 그침 알림 중심 소개
- H1 홈: 핵심 검증 카드 우선 노출
- G1/G2: 목적지 목록과 출발 준비 판단
- H5: 강수 타임라인과 비 그침 알림 토글
- H4/C1/C4: 우산/코디 보조 판단
- A2/A3/A4: 저장/알림 확장 전 계정 gate
- M2/M3/R1~R4: 권한, 설정, 정책 접근

### 제외

- ON Square 정식 소셜 루프
- Weather Note/Reaction/CP/아이템샵
- W1~W4 커뮤니티 날씨 제보
- G3/G4/G5 해외/도보/AI 여행 플래너
- G6 구독 결제
- AdMob 실제 광고 송출

---

## 5. MVP 1 범위

MVP 1은 소수 사용자에게 배포해 실제 반복 사용 여부를 확인한다.
UI 추가보다 알림 신뢰성, 목적지 저장, 권한 복구를 우선한다.

### 포함

- 목적지별 출발 권장 시간 저장
- 비 시작/그침 알림 조건 저장
- 푸시 알림 수신, 열람, 딥링크
- 알림 실패/권한 거부/fallback 상태 문구
- 테스트 사용자 피드백 기록 양식

### 측정

- 첫 실행 후 핵심 기능 이해 여부
- 출발 알림이 실제 외출 판단에 도움이 되는지
- 비 그침 알림을 다시 켜고 싶은지
- 알림이 제때 오지 않았을 때 이탈하는지
- 코디/우산 보조 기능이 핵심 판단을 방해하지 않는지

## 6. 확장 보류 기준

아래 기능은 MVP 사용 지표가 나오기 전까지 개발 우선순위를 낮춘다.

| 기능 | 보류 이유 | 재개 조건 |
|---|---|---|
| 소셜/ON Square | 핵심 날씨 판단과 별도 루프 | MVP 재방문 지표가 충분하고 커뮤니티 필요성이 확인될 때 |
| CP 경제/캐릭터 시스템 | 운영/밸런스 비용 큼 | 리텐션 개선 실험으로 필요성이 확인될 때 |
| 도보/해외여행 플래너 | 데이터/API/일정 UX 범위 큼 | 목적지 날씨 비교 사용률이 높을 때 |
| AI 장소 추천 | 비용과 품질 검증 필요 | 룰 기반 장소 추천으로 수요 확인 후 |
| 광고/구독 결제 | 초기 사용성 검증을 흐림 | MVP 핵심 기능 재방문이 확인된 뒤 |
